# Performance

## Metrics

The testbench reports one performance line per GEMM case:

- total cycles from start to `done`
- total scalar MAC operations, `M * N * K`
- effective MACs per cycle, `mac_ops / cycles`
- theoretical peak MACs per cycle, `TILE_M * TILE_N`
- utilization estimate, `effective / peak`

These are simulation-level cycle metrics only. No synthesis, place-and-route,
timing, power, or FPGA resource results are claimed unless the Vivado flow is
run separately.

## Methodology

The current core intentionally uses a simple memory schedule:

- one A scratchpad read and one B scratchpad read per load cycle
- explicit systolic drain cycles
- explicit C accumulator capture
- one C scratchpad write per store cycle

This makes the control behavior easy to verify, but it leaves performance on
the table. Double-buffering and wider scratchpad/streaming interfaces would be
the natural next optimizations.
